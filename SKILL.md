# OpenHanako 优化大师 v2.2

---

## 概述

优化大师提供**双通道**执行模式：

- 🔵 **手动触发**：说「优化大师」或触发词，展示菜单选择
- 🔵 **定时自动化**：初始化时引导创建定时任务，后台静默执行

两条通道共享同一套规则（优化规则手册 + 文件组织规范），任何规则更新自动同步到两个通道。

---

## 手动入口

你说「优化大师」或匹配触发词时，展示菜单：

```
⚙ 优化大师 v2.2 — 请选择：

  说「初始化」       首次建立目录、规则文档 + 设置定时任务
  说「工作文件夹」   归类文件 + 扫描缓存、会话、临时文件 + 关联同步
  说「插件技能」     扫描技能 / 插件 / 开发规范
  说「系统运行」     扫描经验与记忆 / 配置 / 运行时缓存
  说「全部」        一次性扫描所有
  说「任务状态」     查看定时自动化任务运行状态
  说「退出」        结束
```

```

       - _tools/参考/重点实战笔记规则.md（已有则跳过）

       - 缺少「项目经验沉淀与复盘」→ 按附录C 模板创建（每2天）

第4步：适配检测
第5步：输出报告
```

## 三大板块扫描内容

### 工作文件夹（覆盖板块 ①②⑤）

**第一步：预归类**（文件按类型归位）
扫描工作区根目录白名单之外的文件，按【归类映射表】匹配 → 列出建议去向 → 执行移动。
详见附录B。

**第二步：规则项扫描**
- ① 工作区：根目录越界、项目待归位、`_temp/` 积压
- ② 缓存：日志（保留3天）、检查点（保留10个）、图片缓存（7天前）
- ⑤ 会话：空目录清理、重复文件检测、大文件预警（>50MB）
- `_temp/` 缓冲区：截图>7天 / 数据>14天 / 脚本>14天 / 导出>30天 / 一次性和工作区残留>7天 / 项目结项≥14天→提示移入 projects/
- `data/`：备份>7份确认、归档激活提示
- `_tools/`：闲置脚本>90天提示归档

**第三步：关联同步**（见下方 关联同步规则）

### 插件技能（覆盖板块 ③⑥⑧）

- ③ 技能：同名/相似(>70%)/废弃/工作区副本
- ⑥ 插件：孤儿目录/token-tracker>5MB/单目录>50MB/patcher兼容性
- ⑧ 开发：未结项项目/编译残留

### 系统运行（覆盖板块 ④⑦⑨）

- ④ 经验与记忆：pinned超量(>12~15条)/经验库合并(同类>10条)/认知沉淀/辅助提炼
- ⑦ 配置：设置合理性审计（只读）
- ⑨ 运行时：session积压、`.ephemeral/>7天`、`uploads/temp-upload`陈旧文件、缓存总容量>500MB、plugin-backups/>30天

## 关联同步规则

> 每次归类/移动文件后，自动检索并更新关联引用。

### 执行步骤

```
第1步：记录本次所有被移动的文件（来源路径 → 目标路径）
第2步：对每个被移动的文件，检索以下关联源：
       - pinned memory：逐条搜索旧路径关键字 → 匹配则更新
       - 经验库（experience/）：搜索旧路径关键字 → 匹配则重写
       - 思源笔记：调用 siyuan-sisyphus 全文搜索旧路径 → 匹配则记录到检查点（通知用户手动确认）
       - 其他 SKILL.md：grep 所有 SKILL.md 中包含旧路径的引用 → 匹配则更新新路径
       - 项目文档（项目索引.md、开发反思.md）：grep 匹配 → 更新
第3步：所有自动更新写入检查点的「关联更新」小节
第4步：无法自动处理的（思源笔记关联）记录到检查点，下次用户交互时提示
```

### 分级策略

| 关联源 | 自动更新 | 记录通知 |
|--------|----------|----------|
| pinned memory | ✅ 自动 | — |
| 经验库 | ✅ 自动 | — |
| 其他 SKILL.md | ✅ 自动 | — |
| 项目文档（索引/反思） | ✅ 自动 | — |
| 思源笔记 | ❌ 记录新路径 | 📢 通知用户手动确认 |
| MemOS 知识库 | ❌ 记录新路径 | 📢 通知用户手动确认 |

## 核心执行流程

> 选一个板块，走四步。支持连贯操作模式。

### 扫描 → 报告 → 确认执行 → 沉淀（含关联同步）

### 确认交互规则

1. 每次最多列出 5 项待处理项，你逐一回复「做」或「跳过」
2. 可回复「全部做」批量接受剩余项
3. 低风险操作（移动.md、删除临时文件）汇总执行，不逐条确认
4. 中风险操作（移动数据文件、归档项目）给出分组建议批量确认
5. 高风险操作（删除.exe、卸载插件）才逐条确认
6. 随时说「中断」停止当前操作

### 留痕与异常处理

- 操作留痕保存到 `_temp/检查点/{YYYYMMDD}-归类检查点.md`，保留最新 10 份
- 格式：

```markdown
# {日期} 归类检查点

## 操作记录
| 操作时间 | 项目 | 结果 | 备注 |
|----------|------|------|------|
| HH:mm | 归类 笔记.md → _tools/参考/ | ✅ 已移动 | |

## 关联更新
| 关联源 | 旧路径 | 新路径 | 结果 |
|--------|--------|--------|------|
| pinned memory | _temp/xx | _tools/参考/xx | ✅ 已更新 |
| 思源笔记 | _temp/xx | _tools/参考/xx | 📢 待用户确认 |
```

- 扫描结果为空时反馈："工作区很干净 ⭐"
- 「全部」模式中途想中断 → 说「中断」随时停止

### 启发式判断规则（按优先级）

优先匹配文件名关键字，无匹配则询问用户：

| 关键字（包含任意一个） | 归类 |
|-----------------------|------|
| 讲义/笔记/考点/教材/备考/教资/科目/地理/数学/语文/英语 | 备考资料 |
| 简历/身份证/证件/学历/学位/毕业 | 个人文档 |
| 配置/config/settings/setup | 配置文件 |
| 备份/backup/backup- | 备份 |
| 截图/screenshot/snip/capture | 截图 |
| 项目/plugin/skill/app | 项目 |
| 日记/反思/复盘/经验/log | 个人思考 |
| test/测试/试验 | 一次性 |

### 归类映射表

| 文件特征 | 归入目标 |
|---------|---------|
| `.md` 参考文档/笔记 | `_tools/参考/` |
| `.pdf` 备考资料 | `个性化/资料/备考/{判断科目}/` |
| `.pdf` 个人证件 | `个性化/资料/个人文档/` |
| `.py` `.js` `.go` `.mjs` `.ts` 可复用脚本 | `_tools/脚本/{判断功能分类}/` |
| `.ps1` `.bat` `.sh` 可复用脚本 | `_tools/脚本/{判断功能分类}/` |
| `.zip` `.gz` `.7z` `.rar` 压缩包 | `data/backups/` |
| `.png` `.jpg` `.gif` `.webp` `.bmp` 截图/图片 | `个性化/资料/{判断主题}/截图/` |
| `.json` `.csv` `.sql` `.yaml` `.toml` `.env` 数据/配置文件 | `_tools/脚本/数据/` 或 `个性化/资料/` |
| `.exe` `.msi` `.dmg` `.AppImage` 可执行文件 | `data/archives/` |
| `.docx` `.xlsx` `.pptx` 办公文档 | `个性化/资料/个人文档/` |
| `_temp/项目/{项目名}/` | `projects/{分类}/{项目名}-{状态}/`（由规则项扫描触发） |
| `$null` `Thumbs.db` `.DS_Store` `desktop.ini` | 直接删除 |
| `__pycache__/` `node_modules/` `.git/` | 跳过，不扫描 |
| 项目内 `release/` `portable/` `dist/` `build/` | 跳过，不归类 |
| 不确定类型 | `_temp/工作区残留/` |

### 项目内豁免规则

以下模式在项目目录内视为正常，不列入扫描报告：

| 文件/目录 | 在项目内视为 | 在项目外视为 |
|-----------|-------------|-------------|
| `release/` | 构建产物目录 | `data/backups/` |
| `portable/` | 分发目录 | `data/backups/` |
| `dist/`、`build/` | 构建临时输出 | 应清理 |
| `__pycache__/`、`*.pyc` | Python 缓存 | 应清理 |
| `node_modules/` | JS 依赖 | 应清理 |
| `.exe` | 构建产物 | 归类或清理 |

## 附录C：定时任务模板

### 模板1：每日工作区归类

```markdown
# 每日工作区归类

你正在以 hanako 身份执行每日定时归类任务。严格按以下步骤执行，**不要创建新的定时任务**。

---

## 第一步：加载规则

1. 读 `_tools/参考/优化规则手册.md`，获取最新归类规则和生命周期规则
2. 读 `_tools/参考/文件组织规范.md`，确认当前目录结构标准
3. 内化【归类映射表】和项目内豁免规则

## 第二步：预归类

扫描工作区根目录，找出不在标准白名单（_temp/ _tools/ data/ projects/ 个性化/ .hanako/）中的文件。

对每个越界文件按归类映射表判断去向。**低风险操作直接执行，中风险汇总执行。**

## 第三步：规则项扫描

- `_temp/` 缓冲区生命周期检查（截图>7天 / 数据>14天 / 脚本>14天 / 导出>30天 / 一次性>7天 / 工作区残留>7天 / 项目≥14天）
- 缓存清理（检查点保留10份，日志保留3天，图片缓存7天）
- `data/` 备份≤7份，`_tools/` 闲置>90天预警

## 第四步：关联同步

对本次所有移动的文件，按以下顺序更新引用：
1. **pinned memory** — 搜索旧路径关键字，匹配则更新
2. **经验库** — 搜索旧路径，匹配则重写
3. **其他 SKILL.md** — grep 搜索，匹配则更新
4. **项目文档**（项目索引.md/开发反思.md）— grep 搜索，匹配则更新
5. **思源笔记 / MemOS** — 仅记录到检查点，通知用户

## 第五步：沉淀

- 写入 `_temp/检查点/{YYYYMMDD}-归类检查点.md`
- 如有需要用户确认的文件 → notify 通知
- 无异常 → 安静结束
```

### 模板2：项目经验沉淀与复盘

```markdown
# 项目经验沉淀与复盘（每2天）

你正在以 hanako 身份执行定时经验沉淀任务。定位：与每日工作区归类互补——它管文件，你管知识。

## 执行流程

### 第一步：扫描活跃项目
用 find 扫描 projects/ 下最近3天有改动的项目，检查是否已有 开发反思.md 和 项目索引.md。

### 第二步：查近期对话
用 session list + session read（summary 模式）提取与活跃项目相关的对话内容。

### 第三步：交叉分析与提取
提取关键决策、踩坑记录、有效模式、通用方法论。

### 第四步：更新项目反思文档
追加到 projects/{项目}/开发反思.md。

### 第五步：记录可复用经验
用 recall_experience 检查是否已存在，避免重复。新经验写入经验库对应分类。

### 通知策略
- 重要经验 → notify 用户
- 无异常 → 安静结束
```
