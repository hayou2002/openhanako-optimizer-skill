# OpenHanako 优化大师 v2.2

> Hanako Agent Skill：一站式工作区治理工具。让你的工作区从"知道该整理"到"自动保持整洁"。

说「优化」建立规则约束（治本），说「清理」按规则执行清理（治标）。说板块名跑单项扫描。

---

## 快速开始

```bash
# 在 HanaAgent 中安装
install_skill --github_url https://github.com/hayou2002/openhanako-optimizer-skill

# 说完触发词即可使用
「优化大师」    → 展示菜单
「初始化」      → 首次使用建立目录结构与参考文档
「清理」        → 按现有规则执行全盘扫描清理
```

首次使用先跑一次「初始化」建立骨架，然后说「全部」做一次全盘扫描，之后日常说「清理」维护即可。

## 使用说明

### 双模式设计

| 模式 | 触发词 | 作用 | 频率 |
|------|--------|------|------|
| **优化模式**（治本） | 「优化」 | 建立、更新、细化规则约束 | 工作区结构变化时 |
| **清理模式**（治标） | 「清理」 | 按现有规则执行扫描清理 | 日常维护 |

优化模式修改规则手册和目录结构，清理模式只读规则不修改。两者配合实现"规则驱动执行"。

### 菜单与板块

```
⚙ 优化大师 v2.2 — 请选择：

  说「初始化」       首次使用建立目录与规则文档
  说「工作文件夹」   归类文件 + 扫描缓存、会话、临时文件和 data 目录
  说「插件技能」     扫描技能 / 插件 / 开发规范
  说「系统运行」     扫描经验与记忆 / 配置 / 运行时缓存

  说「全部」        一次性扫描所有
  说「退出」        结束
```

### 标准交互流程

每个板块都走四步：

```
说板块名 → 阶段1：扫描 → 阶段2：报告 → 阶段3：确认执行 → 阶段4：沉淀留痕 → 连贯询问
```

### 确认规则

- **低风险**（移动 .md、删除临时文件）：汇总执行，不逐条确认
- **中风险**（移动数据文件、归档项目）：分组建议，批量确认
- **高风险**（删除 .exe、卸载插件）：逐条确认
- 随时可以说「中断」停止当前操作

### 适配检测

每次扫描前自动检测本地环境，命中以下规则时加载对应策略：

```
个性化/资料/备考/           → 备考规则
个性化/联动/思源笔记/       → 思源笔记联动
个性化/联动/记忆助手-MemOS/ → MemOS 规则
个性化/Hanako/             → Hanako 知识库
个性化/事项/{事项}/         → 事项三模协作
```

## 设计原理

### 本地架构设计

优化大师不是独立的工具——它嵌入在 Hanako Agent 的工作区文件体系中，由五层构成：

```
┌─────────────────────────────────────────────────┐
│  第一层：触发与入口（用户 ↔ Agent 对话层）        │
│  「优化大师」→ 菜单 → 选板块 → 确认 → 执行       │
├─────────────────────────────────────────────────┤
│  第二层：执行引擎（SKILL.md）                     │
│  定义板块划分、扫描规则、交互流程、确认逻辑        │
├─────────────────────────────────────────────────┤
│  第三层：规则手册（优化规则手册.md）               │
│  9+1 条规则分类，覆盖工作区/缓存/技能/插件/经验    │
├─────────────────────────────────────────────────┤
│  第四层：规范文档与参考                           │
│  文件组织规范.md / 笔记排版语法规则 / 重点笔记规则 │
├─────────────────────────────────────────────────┤
│  第五层：运行态（检查点/留痕/适配检测）            │
│  _temp/检查点/ 记录每次操作，适配检测感知本地环境  │
└─────────────────────────────────────────────────┘
```

每一层只依赖下层，不跨层调用。Agent 对话层（第一层）发起指令，经 SKILL.md（第二层）按规则手册（第三层）执行，参考规范文档（第四层）做决策，操作结果沉淀到运行态（第五层）。

### 文件体系与联动关系

优化大师涉及的核心文件及其协作方式：

| 文件 | 角色 | 谁维护 |
|------|------|--------|
| `SKILL.md` | 执行引擎：定义板块、流程、触发规则 | 版本迭代时更新 |
| `_tools/参考/优化规则手册.md` | 规则数据库：9+1 条分类规则 | 优化模式时写入/更新 |
| `_tools/参考/文件组织规范.md` | 目录结构标准：温度体系、生命周期 | 优化模式时建立 |
| `_tools/参考/笔记排版语法规则.md` | 思源笔记排版约束 | 初始化时写入 |
| `_tools/参考/重点实战笔记规则.md` | 笔记内容结构标准 | 初始化时写入 |
| `_temp/检查点/{日期}-{板块}.md` | 操作留痕：每次清理的记录 | 清理模式自动写入 |

**联动流程示例——"清理"→"系统运行"：**

```
用户说「系统运行」
  → SKILL.md 匹配板块，启动扫描
    → 读优化规则手册 ④⑦⑨ 条
      → ④ 检查 pinned memory（逐条判断，不卡条数）
      → ④ 检查经验库（同类超 10 条提示合并）
      → ⑦ 审计配置合理性（只读）
      → ⑨ 检查运行时缓存积压
    → 汇总扫描报告
  → 用户确认执行
  → 写入 _temp/检查点/ 留痕
  → 询问"还要处理其他吗"
```

每个板块独立走这套流程，互不干扰。

### 温度体系与生命周期

工作区按访问频率分为五个温度区，优化大师的清理规则基于此设计：

| 温度 | 目录 | 清理策略 |
|------|------|---------|
| 🔥 热 | `_temp/` | 7~30 天超期自动提醒流转 |
| ☀️ 暖 | `_tools/`、`个性化/` | 90 天闲置预警，不自动删 |
| 🌤 常温 | `projects/` | 结项移入存档 |
| ❄️ 冷 | `data/` | 备份保留 7 份，可反向激活 |

清理模式扫描时，对不同温度区执行不同策略——热区严格、暖区宽松、冷区只提醒。

### 双模式运作机制

**优化模式**和**清理模式**共享同一套扫描引擎，区别在于是否写入规则：

```
                 ┌── 读规则 → 发现缺漏 → 写入/更新规则 ──→ 优化模式
用户触发 → 扫描 ──┤
                 └── 读规则 → 按规则执行清理 → 留痕 ─────→ 清理模式
```

- **优化模式**扫描规则手册是否完整、目录结构是否标准、参考文档是否到位，发现缺漏则补充。它修改的是"规则层"（第三层）。
- **清理模式**只读规则手册，不做任何规则修改。按已有规则扫描工作区、归类文件、清理缓存、合并经验。它操作的是"运行态"（第五层）。

这种分离确保：规则变更经过"优化"决策，日常执行走"清理"流水线，不会因为自动清理误改规则。

### 连贯操作与留痕

每次清理自动写入检查点，保留最近 10 份。执行完一个板块后自动询问"还要处理其他吗"，不需要重新进入菜单。
